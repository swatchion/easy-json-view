## 🔄 技术栈调整

### 原计划 vs 实际实现

| 组件 | 原计划 | 实际实现 | 调整原因 |
|------|--------|----------|----------|
| 平台 | 桌面版 (GTK) | Web版 (WASM) | GTK依赖问题 |
| 数据库 | SQLite | localStorage | Web平台限制 |
| 部署 | AppImage | 静态Web应用 | 平台调整 |
| 服务器 | 无 | Python HTTP Server | 开发便利性 |

### 关键技术决策

1. **平台切换**: 从桌面版切换到Web版
   - **原因**: Ubuntu系统缺少GTK开发库，安装过程遇到依赖冲突
   - **影响**: 改用WebAssembly目标，保持Rust开发体验
   - **收益**: 跨平台兼容性更好，部署更简单

2. **存储方案**: 从SQLite切换到localStorage
   - **原因**: Web平台无法直接访问文件系统
   - **影响**: 数据仅在浏览器本地存储
   - **收益**: 无需数据库配置，即开即用

3. **构建流程**: 增加WASM编译步骤
   - **工具链**: cargo + wasm-bindgen + wasm-opt
   - **优化**: Release模式 + WASM优化
   - **结果**: 最终包大小885KB

## 🔧 开发经验总结

### 成功经验
1. **技术选型灵活性**: 遇到平台问题时快速调整技术方案
2. **模块化设计**: 清晰的服务层架构便于功能扩展
3. **测试驱动**: 完整的单元测试保证代码质量
4. **文档先行**: 详细的需求和计划文档指导开发

### 遇到的挑战
1. **平台依赖**: GTK库依赖问题导致桌面版开发受阻
2. **循环依赖**: 模块间依赖关系需要仔细设计
3. **WASM特性**: Dioxus 0.6.3 需要明确启用web特性
4. **类型系统**: Rust严格的类型系统需要精确的类型匹配

### 解决方案
1. **平台切换**: 及时调整到Web平台，保持开发进度
2. **架构重构**: 将共享类型定义移到服务层避免循环依赖
3. **特性配置**: 正确配置Cargo.toml中的平台特性
4. **类型统一**: 统一数据类型定义，避免类型冲突

### 🔢 行号显示功能实现 (2025-09-29)

#### 功能特性
- **双栏布局**: 左侧行号栏（灰色背景）+ 右侧内容栏，提供清晰的视觉分离
- **智能计算**: 根据格式化后的 JSON 内容自动计算行数并生成对应行号
- **开关控制**: 在顶部格式化配置区域提供 Toggle 开关，默认启用行号显示
- **实时切换**: 用户可以随时开启或关闭行号显示，无需重新格式化
- **专业外观**: 类似代码编辑器的行号显示体验，使用等宽字体和右对齐

#### 技术实现
- **状态管理**: 在 `AppState` 中新增 `show_line_numbers: bool` 字段，默认值为 `true`
- **条件渲染**: 根据开关状态选择渲染带行号或不带行号的输出布局
- **行号生成**: 使用 `output_content.lines().count().max(1)` 精确计算行数
- **布局设计**: 使用 Flexbox 实现行号列（固定宽度）和内容列（自适应宽度）的并排显示
- **UTF-8 安全**: 修复了 `content_preview()` 方法的字符边界问题，确保中文等多字节字符的正确处理

#### 用户体验改进

- **专业体验**: 提供类似专业代码编辑器的查看体验
- **灵活配置**: 支持根据个人喜好选择是否显示行号
